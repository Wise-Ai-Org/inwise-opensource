## Codebase Patterns
- Main-process tests export `run()` + `if (require.main === module) { run().catch(...) }`; run via `node dist/main/<file>.test.js` after `npm run build:main`
- Inject in-memory NeDB Datastores into database.ts with `__setTasksDbForTests` / `__setMeetingsDbForTests` / `__setPeopleDbForTests` to test helpers without Electron
- NeDB supports dot-notation `$set`/`$unset` on array indices (e.g. `insights.actionItems.2.dismissed`); use it for atomic field writes instead of read-modify-write
- `tasksDb.findAsync({ archivedAt: null })` is the canonical "live tasks" query
- Action items live on `meeting.insights.actionItems[i]`; lifecycle state (convertedToTaskId, dismissed) must be persisted there, not on separate projections, or `getPerson()` will always re-render them as pending
- IPC flow: `ipcMain.handle('domain:action', ...)` in main.ts → expose on `inwiseAPI` in preload.ts → wrap in `api` object in src/renderer/api.ts → consumed by components
- Task schema is strict (createTask in database.ts sets `source`, `approval`, `provenance`, `snoozedAt`, `snoozedReason`, `likelyDone` etc.); new task-producing helpers must set the same shape or downstream queries break

- **Learnings for future iterations:**
  - The prior bug was subtle: the mapping at `database.ts:464` hardcoded `convertedToTaskId: null` when reading insights, so even if main.ts had written the back-reference, `getPerson()` would overwrite it in memory. When fixing a "lifecycle state doesn't stick" bug, check the projection layer, not just the writer.
  - NeDB dot-notation `$set: { 'insights.actionItems.${i}.convertedToTaskId': taskId }` works for both setting new fields and updating existing ones — no need to fetch-then-update.
  - `getPerson()` currently fetches `tasksDb.findAsync({ archivedAt: null })` twice (once for the join I added, once for stale-task nudges). Left as-is for this story to keep the change minimal; future iterations could merge them.
  - Summary counts `summary.activeActionItems` / `summary.doneActionItems` are new fields — downstream components that destructure `summary` should be treated as additive consumers.

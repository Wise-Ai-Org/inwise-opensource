## Codebase Patterns
- Config is a single electron-store instance in `src/main/config.ts`. `Config` interface doubles as the schema; `defaults` must list every field (electron-store will not materialize missing fields).
- `getConfig()` returns the whole store snapshot; `setConfig(partial)` writes per-key. Prefer narrow, named helpers (e.g. `addCalendar`, `setSelfEmails`) over dumping partials through `setConfig` so call sites stay readable.
- App start is a single `app.whenReady().then(() => { initDatabase(); ...; calendarWatcher.start(); })` block in `src/main/main.ts`. One-time migrations belong immediately after `initDatabase()` and before any consumer (watcher, windows) reads the migrated fields.
- Logging in main uses the local `log(level, message, detail?)` helper from `src/main/logger.ts` ‚Äî don't introduce `console.*` in main. Levels are limited to `'info' | 'warn' | 'error'` (no `'debug'`).
- Main typecheck: `npx tsc -p tsconfig.main.json --noEmit`. Renderer typecheck: `npx tsc -p tsconfig.renderer.json --noEmit`. Project has no configured test runner, so "tests pass" means the typechecks stay green.
- Deprecated-but-kept fields: annotate with `/** @deprecated */` on the interface and leave the default in place so existing stores still load; only the migration path should read them.

A third meeting-starting while a conflict is pending gets logged at `warn` and dropped ‚Äî a binary modal can't resolve three-way without UX for it.
- `handleMeetingConflict(active, incoming)` serializes both events (Date ‚Üí epoch ms to mirror the existing `calendar:events` channel format), sends IPC `meeting:conflict` to mainWindow, starts a 30s auto-select timer via `pickConflictWinner()`. Uses `isSelf` from US-003 ‚Äî reads `getConfig().selfEmails` + `userName` so users with empty self-config still get a reasonable result (falls through to startTime tiebreaker).

- **Learnings for future iterations:**
  - Date ‚Üí epoch ms serialization is the house convention for main‚Üírenderer event payloads; `calendar:events` does it, and the conflict payload mirrors that shape. If you add a new channel carrying a `Date`, convert at the boundary so the renderer doesn't get `Date`-ish objects back from structured clone (they become plain `Object` with no methods).
  - The conflict flow's "don't re-start recording if winner was already active" check is load-bearing. Without it, picking the already-active meeting would call `createOverlayWindow(winner.title, winner.id)` which reuses the existing overlay but re-sends `recording:start` ‚Äî harmless-looking but re-fires `badge:show` and resets audio-health debounce timers. Always compare `winner.id === active.id` before short-circuiting.
  - Electron `Notification` accepts an `actions` array but only macOS renders them as buttons natively; on Windows they are silently ignored. Always log the decision on the main-process side so reason-tracking doesn't depend on notification action-click behavior.
  - For a countdown in a modal, drive it from a `setInterval` that compares `Date.now()` to a captured `startedAt` ‚Äî don't decrement a counter state each tick. If the modal unmounts and remounts mid-conflict, the counter-based approach drifts or resets; the timestamp-based approach always shows the true remaining time.
  - When a component receives IPC events, always pair `api.on(channel, handler)` with `api.off(channel, handler)` in the cleanup function AND store a reference to the same handler. The preload's `off` uses `removeListener` which requires identity match, so inline arrow functions passed directly would leak.
  - `closeOnOverlayClick={false} closeOnEsc={false}` on the conflict modal is intentional: the main process is the source of truth for conflict resolution. If the user could dismiss the modal without picking, we'd get a state where the 30s timer still fires but the modal is gone ‚Äî safe but confusing. Forcing a pick (or waiting for the timer) keeps main and renderer in sync.

## Codebase Patterns
- Tests live next to source as `X.test.ts` with a `run()` export and `if (require.main === module) run()`. Build via `npx tsc -p tsconfig.main.json`, then run `node dist/main/X.test.js`. No `npm test` script exists.
- `src/main/config.ts` uses electron-store. `electron-store` cannot be imported from compiled JS at arbitrary locations — it must run from within the project's own `dist/main/` so node_modules resolves. Factor pure logic out of store-dependent helpers for unit tests.
- Main-process typecheck: `npx tsc -p tsconfig.main.json --noEmit` (silent = success). Renderer: `npx tsc -p tsconfig.renderer.json --noEmit`.
- `preload.ts` hardcodes an `allowed[]` channel allowlist — every new main→renderer push channel must be added or it's silently dropped.
- Date→epoch-ms is the convention for main→renderer IPC payloads carrying Date objects.
- `mainWindow` hides on close (preventDefault+hide); fallback UI-path checks must verify `mainWindow.isVisible()`, not just `!isDestroyed()`.
- `import { app } from 'electron'` in plain node is safe at load-time (`app` is `undefined`); it only fails when you actually call an `app.*` method. This means tests can import modules that import electron without booting electron.
- NeDB (`@seald-io/nedb`) without a `filename` option is pure in-memory — perfect for unit tests. `{ field: null }` matches documents where the field is null OR missing (Mongo-compatible), so additive schema changes need no migration.
- For testable DB helpers on `database.ts`, use the `__setTasksDbForTests(db)` injection hook. Follow the same pattern if future stories need to exercise meetings/people DBs in unit tests.
- LLM-calling modules follow the `task-completion-inference.ts` pattern: a pure `fooWith(args, llmCaller)` for tests + a production wrapper `foo(args)` that injects the real Claude/OpenAI fetch caller. Do NOT replicate `extractor.ts` (inline `getConfig()` + fetch in the same function) for new stories — it's not unit-testable.
- LLM responses that include IDs MUST be filtered against the set of IDs passed in. Models hallucinate. See `parseInferenceResponse(text, validIds)` as the reference.
- Renderer "session-local" state that must survive re-renders without causing them (e.g. dismissed-id sets) uses `useRef<Set<string>>(new Set())`, not `useState`. Pair with a separate visible-state flag that IS `useState`-driven.
- Pre-check IPCs that gate primary UI (e.g. `welcomeBack:liveMeeting` before WelcomeBack renders) must be fail-open — catch/swallow errors and fall back to the default render path rather than breaking the screen.
- Ask-cards that trigger a side effect AND morph in place (e.g. launchAtStartupOffer → "Done…" confirmation) should own their state in a dedicated sub-component; the parent dispatcher stays dumb and just routes on `ask.kind`. Parent-owned `onLocalDismiss` is reused so `[Not now]` hides the card without dismissing the whole screen.
- Main-process side effects triggered by the renderer (e.g. `app.setLoginItemSettings`) should be wrapped in try/catch and return `{ ok: boolean, error?: string }` so the renderer can present a recovery state without leaking the raw exception.

Learnings for future iterations:
- The **DI pattern** established in `welcome-back.ts` (inject all deps; pure function is the testable surface) was reused here with an extra twist: the LLM call is abstracted as a `LlmCaller` function type, and tests provide mock callers. This is the cleanest pattern in this codebase for anything that makes an external API call — future stories that call an LLM should follow `task-completion-inference.ts` as the reference (NOT `extractor.ts`, which hard-codes `getConfig()` inline).
- **LLM-returned IDs must be validated against a known set.** Models hallucinate IDs. The `validTaskIds` Set filter in `parseInferenceResponse` is non-negotiable — without it, a typo'd ID in the model's response would fail silently at `markLikelyDone` (NeDB `updateAsync` with a non-matching `_id` is a no-op) but corrupt the log line counts.
- **LLM errors in the pipeline must be caught at the boundary.** The pipeline has its own try/catch around the inference step AND the inference function catches internally. Defense-in-depth — a failure in this step cannot abort the pipeline because `pipeline:done` is what triggers the overlay close + toast.
- `markLikelyDone` is **strictly additive** — it does not touch `status`. The PRD is emphatic: never auto-close. Only `confirmLikelyDone` is allowed to set `status='done'`. If a future story adds more "inferred state" fields (e.g., `likelyBlocked`), follow the same 3-helper pattern: `markX`, `confirmX`, `rejectX`, where `markX` NEVER mutates user-owned fields.
- The **push channel `tasks:likely-done-updated`** is sent with no payload — the renderer just knows to re-fetch. If we ever need to tell the UI which specific task was flagged, extend the payload shape; but for now, the simple "something changed, refetch" pattern aligns with how `tasks:reprioritized` already works.
- The PRD's "only evaluate transcripts within 24h" clause is **guaranteed by position in the pipeline** — `runRecordingPipeline` runs seconds after meeting creation, so the transcript is always fresh. No explicit date-check needed in code. If inference is ever run against historical meetings (e.g., a backfill job), that code path needs its own 24h guard.
- Chakra UI's `Button` inside a draggable `Card` needs `e.stopPropagation()` on onClick — otherwise the drag listeners on the card swallow the click. This pattern was already used for the archive/delete buttons; the Done?/No buttons follow the same idiom.
- When adding new `TaskCard` or `TaskColumn` props, **both parent and child must accept them as optional** so that the DragOverlay `<TaskCard>` (rendered outside the Column) doesn't need to plumb every prop. The DragOverlay's TaskCard is a static snapshot — pills/actions won't fire there.

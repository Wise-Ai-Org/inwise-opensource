## Codebase Patterns
- Tests live next to source as `X.test.ts` with a `run()` export and `if (require.main === module) run()`. Build via `npx tsc -p tsconfig.main.json`, then run `node dist/main/X.test.js`. No `npm test` script exists.
- `src/main/config.ts` uses electron-store. `electron-store` cannot be imported from compiled JS at arbitrary locations — it must run from within the project's own `dist/main/` so node_modules resolves. Factor pure logic out of store-dependent helpers for unit tests.
- Main-process typecheck: `npx tsc -p tsconfig.main.json --noEmit` (silent = success). Renderer: `npx tsc -p tsconfig.renderer.json --noEmit`.
- `preload.ts` hardcodes an `allowed[]` channel allowlist — every new main→renderer push channel must be added or it's silently dropped.
- Date→epoch-ms is the convention for main→renderer IPC payloads carrying Date objects.
- `mainWindow` hides on close (preventDefault+hide); fallback UI-path checks must verify `mainWindow.isVisible()`, not just `!isDestroyed()`.
- `import { app } from 'electron'` in plain node is safe at load-time (`app` is `undefined`); it only fails when you actually call an `app.*` method. This means tests can import modules that import electron without booting electron.
- NeDB (`@seald-io/nedb`) without a `filename` option is pure in-memory — perfect for unit tests. `{ field: null }` matches documents where the field is null OR missing (Mongo-compatible), so additive schema changes need no migration.
- For testable DB helpers on `database.ts`, use the `__setTasksDbForTests(db)` injection hook. Follow the same pattern if future stories need to exercise meetings/people DBs in unit tests.

Learnings for future iterations:
- This codebase's task "active" state is `status === 'todo'`, not literally `'active'`. The PRD used 'active' generically; implementation should treat them as synonymous. If a later story needs to distinguish (e.g., 'in_progress' as a separate active state), update the filter.
- `high` AND `critical` priorities both skip the sweep. PRD only spelled out `priority !== 'high'`, but `critical` is strictly more important than `high`, so excluding it is the obvious-intent interpretation. Documented here so future reviewers don't think it was overreach.
- The sweep does NOT touch `lastMentionedAt` for untouched tasks — it reads it only. US-007 (task completion inference) is where `lastMentionedAt` should start being written by the transcript-processing pipeline.
- `lastSweepResult` is a singleton in module state — fine for Electron main process, but if we ever need per-window isolation or SSR, this needs to move to a store. Not currently a concern.
- The whenReady `setTimeout(5_000)` delay is a soft fence — it is NOT synchronised with `calendarWatcher`'s first `events-updated` emission. If US-008's live-meeting suppression needs the sweep result to be guaranteed-present before rendering, promote the sweep call to listen on `calendarWatcher.once('events-updated', ...)` instead.
- Tests build tasks via `createTask()` then backdate with `updateTask(id, { updatedAt: iso(daysAgo), ... })` because `createTask` stamps `updatedAt` to now. This is the standard pattern for time-sensitive DB tests in this codebase.
